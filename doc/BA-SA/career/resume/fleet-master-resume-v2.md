## Вариант 1: Развёрнутый (выделение Notification Service из монолита)

**Проект**: FleetMaster Pro — платформа управления автопарком (fleet management, SaaS)  
**Роль**: Business Analyst / System Analyst  
**Задача**: сквозное сопровождение **выделения Notification Service из монолита** (as-is → контракты → миграция → legacy off)  
**Методология**: Agile (Scrum), DDD (Domain-Driven Design)

**Обязанности:**

- Инвентаризация **as-is** в монолите: триггеры уведомлений, каналы, данные, неявные правила; единый реестр и список пробелов знаний.
- Согласование **bounded context** и стратегии миграции с архитектором и владельцами доменов (feature flag, поэтапное отключение legacy, перенос данных).
- **Parity baseline** и продуктовая **gap**-дельта: user stories и acceptance criteria, в том числе сценарии миграции и приёмки cutover.
- **Event Mapping Matrix** с колонкой «было в монолите»: привязка legacy-триггеров к доменным событиям, каналам, шаблонам, дедупликации.
- Системное проектирование целевого сервиса: контекст и sequence, REST API для фронта, AsyncAPI для потребителей событий, NFR на границе микросервиса.
- Описание **миграции и legacy off**: перенос in-app истории или согласованные альтернативы, критерии приёмки, риски и откат.
- Пакет артефактов для разработки и **скоординированный бэклог** (изменения монолита + новый сервис), Definition of Done на уровне всего выделения.

**5 ключевых достижений:**

1. Снял неопределённость по legacy: собрал as-is по сценариям уведомлений в монолите и закрыл пробелы через интервью и разбор кода — единая база для команды до активной разработки выноса.
2. Снизил риск регресса: ввёл **parity checklist** и явные отличия от as-is с sign-off PO — измеримая приёмка вместо субъективного «как раньше».
3. Ускорил согласование интеграций: зафиксировал ответственность паблишеров (Fleet, Auth, др.) и контракты событий до реализации выделенного сервиса.
4. Сделал cutover управляемым: описал переключение (флаг / dual-run / данные) и **критерии legacy off** — выключение монолитного модуля как формализованное событие.
5. Дал разработке исполнимый бэклог: эпики с задачами на **монолит** (события, удаление прямых вызовов) и на **микросервис** (consumer, каналы, API, наблюдаемость).

**Стек/инструменты**: RabbitMQ, Redis, CloudEvents, OpenAPI, AsyncAPI, PlantUML/Mermaid, Confluence

## Вариант 2: Компактный (2–4 буллита для резюме)

**FleetMaster Pro** — SaaS-платформа управления автопарком | BA/SA | выделение Notification Service из монолита

- Сопроводил выделение уведомлений из монолита: **as-is inventory**, parity/gap, матрица «триггер монолита → доменное событие → каналы», контракты API и событий, критерии **legacy off** и передача в разработку.
- Согласовал event-driven границу с командами домена: публикация событий вместо внутренних вызовов send*, NFR на границе (доставка, идемпотентность, DLQ/retry, наблюдаемость).
- Спроектировал интеграционную и целевую модель: RabbitMQ-топология и маршрутизация, Redis-модель in-app, REST для notification center и preferences, стратегия ошибок (backoff, DLQ).
- Разбил работу на эпики с явным участием **монолита** и нового сервиса; подготовил набор артефактов (требования, OpenAPI/AsyncAPI, диаграммы, риски) для спринтов.

## Вариант 3: Фокус на миграции и микросервисах

**Опыт BA/SA при распиле монолита (notifications):**

- Провёл **strangler-friendly** анализ: карта legacy-триггеров, данных и каналов; определение, что выносится в микросервис, а что остаётся доменными событиями у Fleet/Auth.
- Ввёл **parity checklist** и документированные отклонения от as-is для приёмки без регресса пользовательского опыта.
- Специфицировал **event mapping** с обратной трассировкой к монолиту — основа для отключения старого кода и проверки полноты паблишинга.
- Описал **cutover**: перенос in-app истории / согласованные альтернативы, feature flag, условия полного **legacy off**.
- Согласовал **dual-codebase backlog**: задачи на вырезку вызовов из монолита и на новый Notification Service в одном плане релизов.
