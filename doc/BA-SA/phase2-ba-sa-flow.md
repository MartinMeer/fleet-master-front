## Выделение Notification Service из монолита: флоу бизнес- и системного аналитика

### Постановка задачи

В **монолите уже есть** реализованные уведомления (email, возможно in-app, push и т.д.). Аналитику поставлена задача **от начала до конца** сопроводить выделение этой функции в отдельный микросервис: зафиксировать текущее поведение, согласовать целевую границу и контракты, обеспечить **отсутствие регресса** (или явно оформленные отличия), подготовить приёмку и бэклог для команд монолита и нового сервиса.

Ниже — **один** сквозной флоу. Технические блоки (API, события, NFR, диаграммы) описывают **целевое состояние** после выделения; в каждой фазе указано, что именно аналитик делает в логике «из монолита наружу».

**Ориентир по срокам:** порядка **3–4 недель** (15–24 рабочих дня), если каналов и интеграций много или инвентаризация монолита тяжёлая.

---

### Фаза 1. Рамки, as-is и границы — 3–5 дней

**Цель:** понять *почему* выделяют сервис сейчас, **что монолит делает фактически**, и где проходит **bounded context** нового сервиса.

**1.1. Старт с архитектором и техлидом**

- Что входит в выделяемый сервис, а что остаётся **триггерами** в домене (Fleet, Auth и т.д.): только публикация событий, без бизнес-логики уведомлений в монолите после cutover.
- Принципиальная стратегия миграции: strangler, feature flag, dual-run, окно переноса данных — то, что влияет на требования к наблюдаемости и параллельной работе.
- Зона ответственности аналитика: as-is, parity/gap, контракты, критерии приёмки и отключения legacy.

**1.2. Инвентаризация as-is (ядро фазы)**

Совместно с разработчиками монолита:

- **Карта входов:** пользовательские действия, фоновые джобы, изменения сущностей → какие уведомления уходят (файлы, хендлеры, cron, очереди).
- **Карта данных:** таблицы/поля — шаблоны, логи отправок, read/unread, настройки, подписки на push; есть ли идемпотентность или дедуп только в коде.
- **Карта каналов:** провайдеры email/push, лимиты, ретраи, владельцы секретов.
- **Неявные правила:** тихие часы, отписки, «не слать повторно», язык, фильтры по ролям.

Интервью:

- **PO:** что при переносе **нельзя изменить** с точки зрения пользователя; что можно улучшить осознанно.
- **UX/UI:** как сейчас устроен notification center (если есть) и настройки.
- **Саппорт / операции:** типовые жалобы на уведомления.
- **Команды смежных сервисов:** какие события уже есть или планируются после распила монолита.

**Выход:** документ **As-is inventory** — таблица: *триггер (место в монолите) → канал(ы) → шаблон/текст → данные → ограничения* + список **пробелов** (нельзя восстановить без прогона/логов).

**1.3. Scope целевого сервиса и продуктовая дельта**

Не «доказываем, нужны ли уведомления», а фиксируем **что переносим** и **что добавляем** по сравнению с монолитом.

Пример **в scope** выделения:

- Email (напоминания ТО, просрочки, подтверждения) — *если уже есть в монолите, перенос как parity + улучшения по согласованию*.
- Push (PWA / Web Push), если входит в проект выделения.
- In-app center и preferences — *часто усиливаются при выносе* (отдельный API вместо внутренних вызовов).

Пример **вне scope** (явно фиксируется, чтобы не раздувать выделение):

- SMS, мессенджеры, отдельные сценарии для админов флота — переносятся следующими итерациями.

Бизнес-цели при выделении формулируются как **смесь** архитектурной и продуктовой (пример):

| # | Бизнес-цель | Метрика / критерий |
|---|-------------|-------------------|
| BG-1 | Сохранить или улучшить своевременность напоминаний о ТО | Не хуже as-is по доле пропущенных сценариев; при согласовании — целевое улучшение |
| BG-2 | Независимое развитие и деплой канала уведомлений | Notification Service деплоится без релиза монолита для типовых изменений |
| BG-3 | Сократить время реакции на критические алерты | Сравнимо или лучше as-is; целевой SLA согласовать с PO |
| BG-4 | Предсказуемый контроль пользователя над каналами | Настройки и поведение согласованы с as-is или задокументированы как изменение |

### Фаза 2. Требования: parity, gap, события, NFR — 5–7 дней

**Цель:** превратить as-is и целевую архитектуру в **проверяемые требования**: что обязано совпасть, что меняется, какие события заменяют старые вызовы.

**2.1. Parity baseline и осознанные отличия**

Для каждой строки As-is inventory:

- **Parity:** после cutover поведение для пользователя то же (канал, частота, содержание, условия «не слать»).
- **Отличие допускается:** только с обоснованием и sign-off PO (например другой TTL истории, новый digest).

**2.2. User Stories и AC (продукт + миграция)**

Продуктовые истории описывают **целевой** сервис (их можно трактовать как регрессионную спецификацию там, где раньше всё жило в монолите). Дополнительно — истории именно про выделение:

```
US-M1: Как пользователь существующей системы, я хочу, чтобы после переключения на новый сервис
       мои уведомления по сценариям из списка parity вели себя эквивалентно as-is,
       чтобы не потерять критичные напоминания и алерты.

  AC-M1.1: По parity checklist все отмеченные сценарии пройдены в UAT
  AC-M1.2: Нет необоснованных дублей при at-least-once доставке событий (идемпотентность)

US-M2: Как владелец данных, я хочу, чтобы история in-app за согласованный период
       была перенесена или доступна по согласованной стратегии (перенос / обрезка / fresh start),
       чтобы не было «пустого» центра без предупреждения.

  AC-M2.1: Стратегия миграции истории задокументирована и принята PO
  AC-M2.2: Пользователь видит ожидаемое состояние центра в день cutover
```

Пример продуктовых US в терминах **уже событийной** модели (после выделения):

```
US-1: Как владелец автомобиля, я хочу получать email за 7 дней до планового ТО,
      чтобы успеть записаться на сервис.

  AC-1.1: Email отправляется при получении события maintenance.due_soon
  AC-1.2: Email содержит: название ТО, авто (марка/модель), дату, ссылку на план
  AC-1.3: Email НЕ отправляется, если пользователь отключил email-канал
  AC-1.4: Email НЕ отправляется повторно для того же maintenance_plan.id
  
US-2: Как пользователь, я хочу получать push-уведомление при критическом алерте,
      чтобы немедленно среагировать.

  AC-2.1: Push отправляется при получении alert.created с priority=critical
  AC-2.2: Push содержит: краткое описание, марку/модель авто
  AC-2.3: Нажатие на push открывает экран алерта в приложении
  AC-2.4: Push НЕ отправляется в период do-not-disturb пользователя

US-3: Как пользователь, я хочу видеть все уведомления в notification center в приложении,
      чтобы не пропустить ничего важного.

  AC-3.1: Все уведомления сохраняются in-app независимо от канала
  AC-3.2: Уведомления отмечаются как read/unread
  AC-3.3: Отображается badge-счётчик непрочитанных на иконке колокольчика
  AC-3.4: Уведомления хранятся 90 дней, затем архивируются (или иной срок по parity)

US-4: Как пользователь, я хочу управлять notification preferences,
      чтобы получать только нужные уведомления удобным способом.

  AC-4.1: Настройки доступны в /settings/notifications
  AC-4.2: Можно включить/выключить: email, push, in-app — для каждого типа события
  AC-4.3: Можно задать "тихие часы" (do-not-disturb window)
  AC-4.4: Значения по умолчанию согласованы с as-is или явно изменены с sign-off PO
```

**2.3. Event Mapping Matrix (с привязкой к монолиту)**

Ключевая работа аналитика: каждая строка **должна** ответить, *что в монолите сегодня порождало уведомление* и *какое событие это заменяет*.

| Было в монолите (триггер / модуль) | Входящее событие | Канал(ы) | Шаблон | Приоритет | Дедупликация |
|---|---|---|---|---|---|
| *(заполняется из as-is)* | `maintenance.due_soon` | Email + In-app | `reminder_maintenance` | Normal | По `plan_id` + окно |
| *…* | `alert.created` (critical) | Push + Email + In-app | `critical_alert` | Urgent | По `alert_id` |
| *…* | `alert.created` (medium/low) | Email + In-app | `alert_info` | Normal | По `alert_id` |
| *…* | `service_record.created` | Email + In-app | `service_confirmation` | Low | По `record_id` |
| *…* | `maintenance_plan.overdue` | Email + Push + In-app | `overdue_escalation` | High | По `plan_id` + окно |
| *…* | `user.registered` | Email | `welcome_email` | Low | По `user_id` |

**2.4. Нефункциональные требования (NFR)**

На границе микросервиса NFR становятся явными (очередь, сеть, отдельный деплой):

| # | Категория | Требование |
|---|-----------|------------|
| NFR-1 | Availability | 99.0% (допустима задержка отправки, но не потеря) |
| NFR-2 | Latency | Event-to-delivery: email < 5 мин, push < 30 сек, in-app < 10 сек |
| NFR-3 | Throughput | До 10,000 уведомлений/день на этапе выделения |
| NFR-4 | Durability | Потеря уведомлений = 0 (DLQ + retry) |
| NFR-5 | Idempotency | Повтор события не создаёт дубль уведомления |
| NFR-6 | GDPR/Privacy | Email не хранится в Notification Service — запрос через Auth Service API |
| NFR-7 | Observability | Метрики: sent/failed/retried по каналам, latency p50/p95/p99 |

### Фаза 3. Системный анализ и проектирование — 5–7 дней

**Цель:** описать **как** работает выделенный сервис, **кто** перестаёт вызывать уведомления из монолита и **как** проходит cutover данных.

**3.1. Context Diagram (C4 Level 1)**

После выделения фронт и доменные сервисы не дергают «уведомления внутри монолита», а опираются на **события** и **публичный API** Notification Service:

```
                    ┌───────────────────────────┐
                    │        Frontend           │
                    │  (Notification Center UI) │
                    └────────────┬──────────────┘
                                 │ REST (GET /notifications, PATCH read)
                                 ↓
┌──────────────┐    ┌───────────────────────────┐    ┌──────────────────┐
│ Fleet Service│───→│   Notification Service    │───→│   Auth Service   │
│ (publisher)  │ MQ │                           │REST│ (user email/name)│
└──────────────┘    │  • Event consumer         │    └──────────────────┘
                    │  • Channel router          │
┌──────────────┐    │  • Template engine         │    ┌──────────────────┐
│ Auth Service │───→│  • Delivery dispatcher     │───→│   SendGrid / SES │
│ (publisher)  │ MQ │  • Preference manager      │SMTP│   (email)        │
└──────────────┘    │  • In-app store             │    └──────────────────┘
                    └────────────┬──────────────┘

                                 └───────────────────→│   Web Push API   │
                                              Push    │   (browser push) │
                                                      └──────────────────┘
```

**3.2. Контракты на границе**

- **Publishers:** кто обязан эмитить какие события после вырезки логики из монолита; минимальный payload (ids, приоритет, correlation, без лишнего PII).
- **Синхронный API:** эндпоинты, которые фронт переносит с монолита на новый сервис (ниже — референс).
- **Потребители:** в идеале только Notification Service; обратные вызовы в монолит фиксируются как долг и план отказа.

**3.3. API-контракт (REST endpoints)**

```
Notification Service API (Port 3003):

       Query: ?unread=true&limit=20&offset=0
       Response: { items: [...], unread_count: 5, total: 42 }

PATCH  /notifications/{id}/read     → Отметить как прочитанное

# --- Preferences ---
GET    /notifications/preferences   → Получить настройки пользователя
PUT    /notifications/preferences   → Обновить настройки

       Preferences schema:
       {
         "channels": {
           "email": true,
           "push": true,
           "in_app": true
         },
         "event_settings": {
           "maintenance_reminder": { "email": true, "push": false },
           "critical_alert":      { "email": true, "push": true },

# --- Health ---

```

**3.5. Data Model (целевое хранилище сервиса)**

```
Redis Key Design:

# In-app notifications (sorted set by timestamp)
notifications:{userId}            → ZSET { notifId: timestamp }
notification:{notifId}            → HASH { type, title, body, read, carId, link, createdAt }

# User preferences
preferences:{userId}              → HASH (serialized JSON preferences)

# Idempotency (TTL = 24h)
idempotent:{eventType}:{entityId} → "1" (with TTL)

# Push subscriptions
push:{userId}                     → SET { endpoint1, endpoint2, ... }

# Delivery log (last 30 days, for debugging)
delivery_log:{userId}             → LIST { serialized delivery records }
```

**3.6. Миграция и отключение монолита (обязательный блок для аналитика)**

Зафиксировать в артефактах:

- Перенос **истории in-app** из БД монолита (или согласованный fresh start / обрезка).
- Период **dual-write / feature flag**: кто пишет в старое хранилище, кто в новое; момент переключения чтения фронта.
- Условие **legacy off:** монолит больше не отправляет почту/push/in-app по сценариям из matrix; код помечен на удаление.
- **ADR** по спорным решениям (Redis vs Postgres для in-app, стратегия переноса preferences).

**3.7. Error Handling & Retry Strategy**

| Ошибка | Стратегия | Максимум попыток | Интервал |
|--------|-----------|------------------|----------|
| Email provider (SendGrid) 5xx | Exponential backoff retry | 5 | 1s → 2s → 4s → 8s → 16s |
| Email provider 429 (rate limit) | Delay + retry | 3 | Respect Retry-After header |
| Auth Service unavailable | Retry, затем defer | 3 | 2s → 4s → 8s, затем → DLQ |
| Push endpoint expired | Remove subscription, log | 1 | Immediate cleanup |
| Redis unavailable | Circuit breaker, queue in memory | N/A | 5s health check interval |
| Unrecoverable (bad payload) | Send to DLQ, alert ops team | 0 | Immediate |

### Фаза 5. Передача в разработку — 1–2 дня

**Цель:** единый бэклог для **двух** трасс: новый микросервис и изменения монолита/паблишеров.

**5.1. Выходные артефакты аналитика**

| # | Артефакт | Назначение |
|---|----------|------------|
| 1 | **As-is inventory** | Единая правда о монолите до вырезки |
| 2 | **BRD / пакет требований** | Scope, цели, US, parity, gap |
| 3 | **Parity checklist** | Приёмка эквивалентности |
| 4 | **Event Contract** | AsyncAPI — события вместо внутренних вызовов |
| 5 | **API Contract** | OpenAPI — эндпоинты для фронта |
| 6 | **Data Model + план миграции** | Хранилище сервиса + перенос / обрезка |
| 7 | **Sequence / context diagrams** | Согласование с командами |
| 8 | **NFR + SLA** | Эксплуатация границы |
| 9 | **ADR** | Решения по миграции и стеку |
| 10 | **Risk assessment** | Регресс, дубли, откат |

**5.2. Definition of Done выделения**

Задача аналитика считается закрытой по выделению не при «черновике спеки», а когда разработка может исполнять критерий:

- Parity checklist пройден (или отличия явно приняты PO).
- Контракты событий и API согласованы; паблишеры знают, **что убрать из монолита**.
- Зафиксировано условие **legacy off** и ответственные за флаг / cutover.

### Визуализация сквозного флоу

```
 Фаза 1                 Фаза 2                    Фаза 3                      Фаза 4              Фаза 5
 РАМКИ + AS-IS          ТРЕБОВАНИЯ                  ПРОЕКТИРОВАНИЕ               ВАЛИДАЦИЯ           ПЕРЕДАЧА
 3–5 дней               5–7 дней                   5–7 дней                     2–4 дня             1–2 дня
 ──────────             ──────────                 ──────────                   ──────────          ──────────

 Bounded context    →   Parity + gap            →  Context + contracts       →  Review + cutover  →  Артефакты
 As-is inventory        US + US-M*                  API + AsyncAPI               Parity sign-off       + бэклог
 Scope / цели           Event matrix (+ монолит)    Миграция + legacy off       Риски               монолит + MS
 Стейкхолдеры           NFR                         Sequence + data model        Open questions      DoD выделения
```

**Итог:** один документ ведёт аналитика от задачи «в монолите уже есть уведомления» до передачи в разработку с **явной** линией миграции, parity и отключения legacy.
